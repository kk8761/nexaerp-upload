# NexaERP Project Structure

## Overview

This document describes the monolithic TypeScript project structure for the NexaERP enterprise platform.

## Layer Architecture

### 1. Controllers Layer (`src/controllers/`)

**Purpose**: Handle HTTP requests and responses

**Responsibilities**:
- Parse request parameters, query strings, and body
- Validate input data
- Call appropriate service methods
- Format and send responses
- Handle errors and pass to error middleware

**Example**:
```typescript
// UserController.ts
class UserController {
  async createUser(req: Request, res: Response, next: NextFunction) {
    // 1. Extract data from request
    // 2. Validate input
    // 3. Call UserService
    // 4. Send response
  }
}
```

### 2. Services Layer (`src/services/`)

**Purpose**: Implement business logic

**Responsibilities**:
- Execute business rules
- Coordinate between multiple models
- Perform complex calculations
- Handle transactions
- Return data to controllers

**Example**:
```typescript
// UserService.ts
class UserService {
  async createUser(data: CreateUserDTO): Promise<IUser> {
    // 1. Hash password
    // 2. Create user in database
    // 3. Return user object
  }
}
```

### 3. Models Layer (`src/models/`)

**Purpose**: Define data structures and database schemas

**Responsibilities**:
- Define Mongoose schemas
- Set up validation rules
- Define relationships
- Add instance and static methods
- Configure indexes

**Example**:
```typescript
// User.ts
const userSchema = new Schema<IUser>({
  email: { type: String, required: true, unique: true },
  password: { type: String, required: true },
  // ... other fields
});

export default mongoose.model<IUser>('User', userSchema);
```

### 4. Middleware Layer (`src/middleware/`)

**Purpose**: Process requests before reaching controllers

**Responsibilities**:
- Authentication (JWT verification)
- Authorization (role-based access)
- Request validation
- Logging
- Error handling

**Example**:
```typescript
// auth.ts
export const authenticate = (req, res, next) => {
  // Verify JWT token
  // Attach user to request
  // Call next() or send error
};
```

### 5. Utils Layer (`src/utils/`)

**Purpose**: Provide reusable utility functions

**Responsibilities**:
- Logging utilities
- Response formatters
- Date/time helpers
- Encryption utilities
- File handling

**Example**:
```typescript
// response.ts
export class ResponseHandler {
  static success(res, data, message) { ... }
  static error(res, message, statusCode) { ... }
}
```

## Module Organization Pattern

For each major feature (CRM, Inventory, Manufacturing, etc.), follow this pattern:

```
src/
├── controllers/
│   ├── CRMController.ts
│   ├── InventoryController.ts
│   └── ManufacturingController.ts
│
├── services/
│   ├── CRMService.ts
│   ├── InventoryService.ts
│   └── ManufacturingService.ts
│
└── models/
    ├── Lead.ts
    ├── Opportunity.ts
    ├── Product.ts
    ├── InventoryItem.ts
    ├── BOM.ts
    └── ProductionOrder.ts
```

- Strict mode enabled

- ESLint rules for code quality
- TypeScript-specific rules
- Prettier integration

- Code formatting rules

### package.json
- Project metadata

- NPM scripts for development and production

## NPM Scripts

| Script | Command | Purpose |
|--------|---------|---------|
| `dev` | `nodemon --watch src --ext ts --exec ts-node src/server.ts` | Development with hot reload |
| `build` | `tsc` | Compile TypeScript to JavaScript |
| `build:watch` | `tsc --watch` | Watch mode compilation |
| `start` | `node dist/server.js` | Run production server |
| `lint` | `eslint src/**/*.ts` | Check code quality |
| `lint:fix` | `eslint src/**/*.ts --fix` | Fix linting errors |
| `format` | `prettier --write "src/**/*.ts"` | Format code |
| `format:check` | `prettier --check "src/**/*.ts"` | Check formatting |
| `type-check` | `tsc --noEmit` | Type check without emitting |

## Best Practices

### 1. Type Safety
- Always define interfaces for data structures
- Use strict TypeScript settings
- Avoid `any` type when possible

### 2. Error Handling
- Use try-catch in async functions
- Pass errors to Express error middleware
- Log errors with context

### 3. Code Organization
- One class per file
- Group related functionality
- Keep files under 300 lines

### 4. Naming Conventions
- PascalCase for classes and interfaces
- camelCase for variables and functions
- UPPER_CASE for constants
- Descriptive names (no abbreviations)

### 5. Documentation
- Add JSDoc comments for public APIs
- Document complex business logic
- Keep README up to date
